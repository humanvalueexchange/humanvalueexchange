## 2. Corrections and Clarifications

### 2.1 The jr Architecture — Clarified (addresses #63)

**ADR-006 as written implied:** jr directly connects to the Hermes MCP server as a client (potentially via Tailscale).

**Corrected architecture:**

```
Phase 1 (current): jr → Open WebUI (chat) → Hermes LLM → Hermes MCP Tools
Phase 3+ (future): jr → Tailscale → Hermes Gateway → Hermes LLM → Hermes MCP Tools
```

jr **never calls the MCP server directly.** Jr interacts via natural language in Open WebUI. Hermes (the LLM running in hermes-gateway) calls MCP tools, processes the raw API responses, and presents a summarized, context-appropriate answer back to jr. Jr's 4K context window sees only Hermes's natural language reply — not raw GitHub API payloads.

**Consequence:** Grok #63's concern (large file contents crashing a 4K context consumer) does NOT apply to Phase 1. It WILL apply when jr graduates to an autonomous agent loop with direct MCP access (Phase 3+). When that happens, server-side summarization wrappers and per-client context budgets must be in place first.

**Remaining obligation from #63:** ADR-006 language "enables jr immediately" is accurate for Phase 1 (jr benefits via Hermes LLM mediation), but misleading about jr's autonomous capability. Future phases that give jr direct MCP access must pass a capability gating review before deployment. Grok's recommendation (read-only, bounded tools first; server-side excerpting) applies to Phase 3+ design.

### 2.2 Per-Caller Identity and Attribution — Required (addresses #62)

**Gap identified:** Single shared `X-HVE-API-Key` means all callers are indistinguishable. GitHub mutations (create_issue, comment, vote) carry no caller identity. The ADR's "fully auditable" claim was not satisfied.

**Correction: Attribution is now a Phase 1 gate, not a future enhancement.**

#### Required changes before Phase 1 goes to production:

**A. `X-HVE-Agent-ID` header (new optional-now, required-soon)**

Callers SHOULD include an `X-HVE-Agent-ID` header (e.g. `hermes`, `grok-build`, `copilot-studio`). The server logs this on every tool call. In Phase 2 it becomes mandatory.

**B. Attribution in all GitHub mutations**

All write tools (`create_github_issue`, `comment_github_issue`, `vote_backlog_issue`, `suggest_backlog_issue`) must append caller attribution when an Agent-ID is provided:

```
_Via Hermes MCP — called by {agent_id} at {timestamp}_
```

If no Agent-ID is present, attribute to `hermes` (default / self-use).

**C. Structured tool call logging**

Every MCP tool invocation must be logged to `~/hermes-cfo/logs/mcp-audit.jsonl` with:
```json
{"ts": "ISO8601", "agent_id": "str", "tool": "str", "repo": "str|null", "result_chars": 0, "duration_ms": 0}
```
Args are NOT logged (prevent credential leakage in logs). Tool name, caller, timing, and output size only.

**D. Network exposure review**

Hermes MCP currently binds `0.0.0.0:8765`. This is acceptable IF:
- Firewall rules limit access to LAN (192.168.1.x) and Tailscale interface only
- `HVE_MCP_API_KEY` is always set in production (non-empty = auth enforced)

Hans to confirm firewall posture. Until confirmed: treat hub as LAN-only.

### 2.3 Current State → Target Architecture Migration (addresses #65)

**Gap identified:** Grok Build is currently configured with a direct GitHub PAT (`~/.grok/config.toml: github-hve`) — exactly the anti-pattern ADR-006 rejected. The ADR described the end state but not the migration path.

#### Current state (as of 2026-06-04)

| Agent | GitHub access method | Status |
|---|---|---|
| Hermes | `tools/github_issues.py` (Hermes MCP native tools) | ✅ Hub-native |
| Grok Build | `~/.grok/config.toml github-hve` community server + direct PAT | ⚠️ Tech debt |
| Copilot Studio | None yet (planned via Tailscale Funnel) | 🔲 Not deployed |
| jr | None (Open WebUI chat only) | ✅ No direct access needed |

#### Migration path (Grok #65)

**Phase 1 gate (before Grok migrates):**
1. Hermes GitHub tool surface must cover Grok's primary use cases (issues, files, search, commits — ✅ now done in PR #66)
2. Attribution (#62) must be implemented so Grok's calls are distinguishable from Hermes self-calls
3. Test: Grok calls Hermes endpoint directly and confirms functionality parity for its common workflows

**Migration steps (Grok Build):**
```toml
# ~/.grok/config.toml — AFTER migration
[mcp_servers.hermes-hve]
url = "http://192.168.1.10:8765/mcp"
[mcp_servers.hermes-hve.headers]
X-HVE-API-Key = "${HVE_MCP_API_KEY}"
X-HVE-Agent-ID = "grok-build"
```

Until migration is complete: Grok's direct PAT config is acknowledged tech debt. It is NOT a blocker for Phase 1 deployment to Hermes self-use.

**Note:** Some GitHub operations in Grok's community server surface may not yet be in Hermes (PR review, branch management, etc.). Those operations are tracked in ADR-006 Phases 2 and 3. Grok should maintain dual config (community server + Hermes endpoint) during the migration window.

### 2.4 Deployment Rule — Clarified and Simplified (addresses #64)

**Gap identified:** ADR-006 stated "two manual steps required for every new tool." This created confusion between two distinct services.

#### Two services — two different behaviors

| Service | File | Tool registration | Restart needed? |
|---|---|---|---|
| `hermes-mcp.service` | `mcp/server.py` | **Automatic** via `@mcp.tool()` decorator | On code deploy only |
| `hermes-gateway.service` | `~/.hermes/profiles/main/config.yaml` | **Manual** — add to `tools.include` | On config change only |

The **MCP server** (hermes-mcp.service) auto-registers all `@mcp.tool()` decorated functions at startup. No config file needed.

The **Hermes gateway** (hermes-gateway.service) is the Hermes *agent* process. Its `config.yaml tools.include` list controls which tools Hermes *as an agent* is allowed to call. A new tool added to the MCP server is NOT callable by Hermes until it is also added to `tools.include` and the gateway is restarted.

**Corrected deployment rule:**

```
New tool added to mcp/server.py:
  → sudo systemctl restart hermes-mcp          ← MCP server picks up new tool
  → Edit ~/.hermes/profiles/main/config.yaml   ← Add to tools.include
  → systemctl --user restart hermes-gateway    ← Agent picks up new tool
```

**Phase 2 improvement target:** Eliminate the manual config.yaml step by making `tools.include` dynamic (auto-generated from registered tool names, with an explicit block-list for dangerous/experimental tools). This is tracked as part of #64.

#### Rate-limit and quota monitoring

Absent from Phase 1. Required before Phase 2 (additional API integrations). Target implementation:
- Response header capture (`X-RateLimit-Remaining`, `Retry-After`) in `_request_github_json`
- Warning logged to `mcp-audit.jsonl` when remaining < 20% of limit
- `get_client_context` tool to expose rate-limit status to Hermes for self-throttling

This is tracked in #64 as a Phase 2 prerequisite.

## 4. Acceptance Criteria — GitHub MCP Tools (Phase 1)

Grok validation checklist (per Grok's recommendation in #63 and test plan in `docs/test-plans/github-hub-tools-test-plan.md`):

1. **All 22 unit tests pass** — zero failures, zero skips
2. **Repo allowlist** — all 4 tools reject any repo not in `VALID_REPOS`
3. **Content truncation** — `get_github_file` truncates at exactly 8000 chars
4. **Query scoping** — `search_github_code` scopes to org or repo, never global
5. **Commit message** — `list_github_commits` shows only first line of multi-line messages
6. **Signature deduplication** — `create_github_issue` appends Hermes signature only once
7. **Attribution** — all write tools include `X-HVE-Agent-ID` value in GitHub body (if header present)
8. **Audit log** — every tool call appends one JSON line to `~/hermes-cfo/logs/mcp-audit.jsonl`
9. **No token in error output** — error messages never expose raw token values
10. **MCP registration** — all 4 tools visible in `/.well-known/agent.json` after deployment
